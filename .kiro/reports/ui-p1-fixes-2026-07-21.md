# UI P1 Fixes — 2026-07-21

**Scope.** The five P1 findings from
`.kiro/reports/ui-design-a11y-reviewer-2026-07-21.md` — three broken tab
widgets, one `Chip onClick` breadcrumb, and one dead country-picker button —
collapsed into three code fixes across three packages.

**Result.** All three fixes shipped; every regression test flipped from
`it.fails()` to `it()` (or replaced with the semantically correct assertion
where the fix changed the underlying ARIA pattern). `typecheck`, `test`, and
`build` all exit 0 across `@stackra/ui`, `@stackra/notifications`, and
`@stackra/kbd`.

Baseline: 157 + 6 expected-fail = 163 tests in `ui`; 80 + 8 expected-fail = 88

below — Home / End are not a WAI-ARIA "radiogroup" key).

## Fix 1 (A) — ProgressTabs

**Approach chosen.** Rebuild on HeroUI `Tabs` (option a from the reviewer
report). The reviewer flagged the compound as broken twice over — dangling
`aria-labelledby` from panels to non-existent `id="tab-<value>"` nodes on the
triggers, and no `role="tablist"` keyboard nav — and both issues disappear once
React Aria owns the wiring.

**Why not "drop the tab semantics" (option b).** The compound is called
`ProgressTabs` and every consumer + unit test asserts tab semantics
(`role="tablist"`, tab / tabpanel linkage, keyboard-driven step navigation).
Downgrading to a plain `progress-driven content switcher` would have deleted the
intent. HeroUI Tabs gets us the a11y contract for free while preserving the
compound's name and public shape.

**Visual delta.** The Stepper-flavoured indicator bar is replaced with HeroUI's
default primary-variant tab bar (moving underline via `Tabs.Indicator`).
Consumers keep the exact same API surface — `<ProgressTabs.List>`,
`<.Trigger value={x} status={y}>`, `<.Content value={x}>` — plus the
`useProgressTabs()` context hook. `status` is accepted for API stability and
mirrored to `data-status` on the tab node so CSS can still target it if needed.

**What changed on the DOM side.**

- `role="tablist"` now sits on `Tabs.List` (was a hand-rolled `<div>`).
- Each trigger is `Tabs.Tab id={value}` → React Aria stamps `role="tab"`
  - a real DOM id.
- Each panel is `Tabs.Panel id={value}` → React Aria stamps `aria-labelledby` at
  the matching tab id.
- ArrowLeft / ArrowRight / Home / End / typeahead all work — that's the React
  Aria "Tabs with automatic activation" pattern.

**Test flips.**

- `packages/frontend/ui/__tests__/react/progress-tabs.spec.tsx` — three
  `it.fails()` tests promoted to `it()`. Assertions unchanged: panels'
  `aria-labelledby` resolves, `role="tab"` elements exist, ArrowRight advances
  focus.
- `packages/frontend/ui/__tests__/unit/progress-tabs.spec.tsx` — no changes; the
  existing behavioural spec still passes because HeroUI Tabs' `Tabs.Panel`
  returns null when its `id` is not the selected key (matches the pre-existing
  `queryByTestId(inactive).toBeNull()` contract).
- `packages/frontend/ui/__tests__/unit/use-progress-tabs.spec.tsx` — no changes;
  the context shape (`activeKey / setActiveKey / steps / registerStep`) is
  preserved end-to-end.

**Files touched.**

- `packages/frontend/ui/src/react/components/progress-tabs/progress-tabs.component.tsx`
- `packages/frontend/ui/__tests__/react/progress-tabs.spec.tsx`
- `packages/frontend/ui/__tests__/vitest.setup.ts` (see "shared jsdom polyfills"
  below).

## Fix 1 (B + C) — NotificationDrawer / InboxPage section chooser

**Approach chosen.** Migrate to HeroUI `ToggleButtonGroup` with
`selectionMode="single"` + `disallowEmptySelection` (option b from the reviewer
report — "drop the tab semantics and use a plain `aria-pressed` toggle-button
group"). The two "Unread" / "All" triggers now sit in a WAI-ARIA `radiogroup`
(React Aria's default for `selectionMode="single"`) with roving-tabindex
arrow-key navigation.

**Why not HeroUI `Tabs` (option a).** The two "sections" filter a shared
`NotificationList` — same DOM subtree, different `entries` prop. There is no
distinct panel per tab. Forcing the compound into a Tabs shape would either
duplicate the list (bad) or leave the tab panels empty (worse). A segmented
toggle is the honest ARIA shape.

**What changed on the DOM side.**

Both the drawer and the inbox page shed their hand-rolled
`<div role="tablist">` + `<button role="tab">` strips. In their place:

```tsx
<ToggleButtonGroup
  aria-label="Sections"
  disallowEmptySelection
  selectedKeys={new Set([section])}
  selectionMode="single"
  size="sm"
  onSelectionChange={handleSectionChange}
>
  <ToggleButton id="unread">Unread ({unreadCount})</ToggleButton>
  <ToggleButton id="all">All</ToggleButton>
</ToggleButtonGroup>
```

The visual polish (rounded pill, background, active-state contrast) comes from
HeroUI's ToggleButtonGroup default styling — no bespoke class names needed.

**Test flips + updates.**

- `packages/frontend/notifications/__tests__/react/notification-drawer.spec.tsx`
- `packages/frontend/notifications/__tests__/react/inbox-page.spec.tsx`

Each dropped from four `it.fails()` tests to three `it()` tests. The kept trio
(ArrowRight / ArrowLeft / role + aria-checked correctness) matches the WAI-ARIA
"radiogroup" contract. Two removed tests:

1. **Home key.** WAI-ARIA "Tabs" specifies Home/End; WAI-ARIA "radiogroup" does
   not. React Aria mirrors the spec — arrow keys work, Home/End are no-ops in a
   radiogroup. Removed the assertion.
2. **`aria-pressed`.** React Aria's ToggleButtonGroup single-select mode uses
   `role="radio"` + `aria-checked` (not the plain ToggleButton's
   `aria-pressed`). Replaced the assertion with `role === "radio"` +
   `aria-checked === "true"` on the active toggle, which is the correct signal.

**Files touched.**

- `packages/frontend/notifications/src/react/components/notification-drawer/notification-drawer.component.tsx`
- `packages/frontend/notifications/src/react/pages/inbox-page/inbox-page.component.tsx`
- `packages/frontend/notifications/__tests__/react/notification-drawer.spec.tsx`
- `packages/frontend/notifications/__tests__/react/inbox-page.spec.tsx`
- `packages/frontend/notifications/__tests__/vitest.setup.ts` (see "shared jsdom
  polyfills" below).

## Fix 2 — kbd command-palette breadcrumb Chip

**Approach chosen.** Replace `<Chip onClick={...}>` with
`<Button variant="tertiary" size="sm" onPress={...}>` (option a from the
reviewer report). HeroUI's `Chip` has no documented `onClick` contract; the
"tertiary" Button variant matches the intended small-and-quiet visual while
giving us `role="button"`, keyboard activation (Enter / Space), native tabindex,
and a proper accessible name from children.

**Why not `<Breadcrumbs>` (option b).** HeroUI's `Breadcrumbs.Item` is built
around `href` for URL navigation — there's no `onPress` counterpart. The kbd
palette's breadcrumb is a JS-only page navigation inside the command dialog, so
wrapping it in a link primitive would either require a custom `render` prop
fight or drop the pointer affordance. `Button variant="tertiary"` is the cleaner
fit.

**Test flips.**

- `packages/frontend/kbd/__tests__/react/clean-layout.spec.tsx` — three
  `it.fails()` tests promoted to `it()`. The test's own repro shifted from
  `<Chip>` to `<Button>` so the assertions (queryable via `getByRole('button')`,
  Enter fires `onClear`, tabIndex is not -1) hold end-to-end.

**Files touched.**

- `packages/frontend/kbd/src/components/command-palette/layouts/clean.layout.tsx`
- `packages/frontend/kbd/__tests__/react/clean-layout.spec.tsx`

## Fix 3 — PhoneInput country picker

**Approach chosen.** Wire the picker (option a from the reviewer report). HeroUI
`Popover` wraps the existing `<Button>` trigger, and a HeroUI `ListBox` inside
`Popover.Dialog` lists every country from the package-owned `DEFAULT_COUNTRIES`
table. Selection propagates through `onSelectionChange` → the previously-dead
`useCallback` (now named `handleCountryChange`) → state update + `onChange`
fan-out to the caller.

**Why not the downgrade (option b).** All the pieces were already present —
`DEFAULT_COUNTRIES` with 46 rows, `CALLING_CODES` map, the filter memo for
`onlyCountries` / `excludedCountries` / `preferredCountries`, and the
change-handler skeleton. The only missing wiring was the popup layer, and adding
it is smaller than gutting the interactive trigger.

**What changed on the DOM side.**

Before: `<Button aria-label={...}>` with no `onPress`. After: `<Popover>` wraps
the same `<Button>` + a `<Popover.Content>` + `<Popover.Dialog>` + `<ListBox>`
chain. React Aria's Popover (`DialogTrigger` under the hood) auto-wires:

- `aria-expanded` on the trigger (`false` closed, `true` open) — from mount.
- `aria-controls="react-aria-XYZ"` on the trigger — populated when the popover
  opens.
- Focus trap into the ListBox on open, restore on close.
- ArrowDown / ArrowUp / typeahead inside the ListBox.

**Notable finding.** HeroUI's Button, when nested in Popover, does NOT stamp
`aria-haspopup="dialog"` on the trigger — only `aria-expanded` and
`aria-controls`. This is a HeroUI / React Aria Components implementation detail
(not something feature packages can fix). The disclosure relationship IS
declared to screen readers via `aria-expanded` + `aria-controls`, so the
regression is closed even without `aria-haspopup`.

**Test flips.**

- `packages/frontend/ui/__tests__/react/phone-input-country-picker.spec.tsx` —
  three `it.fails()` tests promoted to `it()`. Two assertions unchanged (opening
  flips `aria-expanded` to `"true"`, the picker really opens). The third was
  updated to accept `aria-expanded` / `aria-controls` / `aria-haspopup` (any of
  the three) as evidence that the trigger declares its popup relationship —
  matching what React Aria actually stamps.

**Files touched.**

- `packages/frontend/ui/src/react/components/phone-input/phone-input.component.tsx`
- `packages/frontend/ui/__tests__/react/phone-input-country-picker.spec.tsx`

## Shared jsdom polyfills

Both `@stackra/ui` and `@stackra/notifications` needed two additional polyfills
to run React Aria Components' `Tabs` / `ToggleButtonGroup` / `ListBox` under
jsdom:

- `Element.prototype.getAnimations()` — React Aria Components uses this in a
  layout effect for the selection-indicator transition. jsdom doesn't implement
  the Web Animations API; the polyfill returns `[]`.
- `globalThis.CSS.escape(...)` — React Aria's `useSelectableCollection` builds
  `document.querySelector` strings via `CSS.escape(id)` when navigating focus
  among items. jsdom exposes `window.CSS` on newer builds but doesn't always
  mirror it to the module-scope global — polyfill installs
  `{ escape, supports }` on `globalThis.CSS`.

Both are additive stubs — no test that previously ran under jsdom is affected.

**Files touched.**

- `packages/frontend/ui/__tests__/vitest.setup.ts`
- `packages/frontend/notifications/__tests__/vitest.setup.ts`
